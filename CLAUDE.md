# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Primodium is a fully on-chain, space-based factory-building game built with **MUD v2** (Lattice) and **Phaser 3**. It's a pnpm monorepo (pnpm 8.x, Node 20.x) with 6 packages.

## Commands

### Full Stack Development

```bash
pnpm dev                # Runs client + core + anvil devnode + local deploy + indexer (via run-pty)
pnpm dev:client         # Client + core watch only (assumes contracts already deployed)
pnpm dev:contracts      # Anvil devnode + local deploy only
```

### Building

```bash
pnpm build              # Build all packages
pnpm build:contracts    # Build contracts only (runs codegen + MUD build + abi-ts)
```

### Contracts (run from `packages/contracts/`)

```bash
forge build && pnpm mud test --skipBuild   # Run all contract tests (how `pnpm test` works)
forge test --match-test testFunctionName   # Run a single contract test
forge test --match-contract ContractName   # Run all tests in a contract
pnpm run build:terrain                     # Regenerate terrain codegen
pnpm run build:prototype                   # Regenerate prototype codegen
```

### Client (run from `packages/client/`)

```bash
pnpm dev         # Vite dev server on http://localhost:3000
pnpm test        # Vitest
```

### Core SDK (run from `packages/core/`)

```bash
pnpm build       # tsup build
pnpm dev         # tsup watch mode
pnpm test        # Vitest
```

### Linting

```bash
# Contracts
cd packages/contracts && pnpm lint         # prettier + solhint

# Client
cd packages/client && pnpm lint            # eslint
cd packages/client && pnpm lint:fix        # eslint --fix
```

### Deployment

```bash
pnpm deploy:local              # Local anvil (skipBuild)
pnpm deploy:caldera            # Caldera L2
pnpm deploy:caldera-sepolia    # Caldera Sepolia testnet
pnpm deploy:redstone-holesky   # Redstone Holesky testnet
```

## Architecture

### Package Dependency Graph

```
client → core, game, engine, assets, contracts
game   → core, engine, assets, contracts
core   → contracts (MUD tables/types)
engine → (standalone Phaser abstraction)
assets → (standalone sprite/audio/vfx assets)
```

### `contracts` — Solidity / MUD v2 (packages/contracts/)

- **Solidity 0.8.24**, Foundry, MUD namespace `Pri_11`
- **Systems** (`src/systems/`): Player-facing entry points. Organized into top-level systems (CombatSystem, SpawnSystem, etc.) and subdirectories (`fleet/`, `building/`, `transfer/`). Subsystems in `subsystems/` have `openAccess: false` (called only by other systems, not directly by players).
- **Libraries** (`src/libraries/`): Stateless logic used by systems (LibCombat, LibBuilding, LibResource, LibUnit, etc.)
- **Codegen** (`codegen/`, `src/codegen/`): Auto-generated from MUD config and TypeScript generators. Do not edit directly. Regenerated by `build:lib` and `build:mud`.
- **Config** (`config/`): Prototype definitions and enum configs consumed by TypeScript codegen (`ts/prototypes/`, `ts/terrain/`)
- **Tables**: Defined in `mud.config.ts`. All game state lives in MUD tables (Position, Home, P*GameConfig, ResourceCount, etc.). `P*` prefix = prototype/config tables.
- **Tests** (`test/`): Foundry tests extending `PrimodiumTest` (from `test/PrimodiumTest.t.sol`), which extends MUD's `MudTest`. Provides pre-configured `world`, user addresses (`alice`, `bob`, `eve`), and resource shortcuts.
- **Key constants** (`src/constants.sol`): `RESOURCE_SCALE = 1e18`, speed/multiplier scales at 100

### `core` — TypeScript SDK (@primodiumxyz/core, packages/core/)

- Published npm package with two entry points: `.` (core) and `./react` (React hooks)
- Provides Tables, Network config, Utils, and Sync functionality
- Uses `@primodiumxyz/reactive-tables` for reactive MUD table state
- RxJS for stream-based state updates, viem for blockchain interaction
- 19+ React hooks (useCore, useAllianceName, useCooldownEnd, etc.)

### `game` — Game Logic (@primodiumxyz/game, packages/game/)

- Phaser 3 scenes: asteroid, command-center, starmap, root, ui
- Game objects and rendering in `src/lib/`
- Consumer API in `src/api/`
- Zustand for game state management

### `engine` — Phaser Engine (@primodiumxyz/engine, packages/engine/)

- Low-level Phaser scene management abstraction
- RxJS + Zustand for engine state
- Used by `game` package, not directly by `client`

### `client` — React Web App (packages/client/)

- React 18 + Vite, Tailwind CSS + DaisyUI
- Wagmi + viem for wallet/chain interaction
- Env vars prefixed with `PRI_` (Vite envPrefix), loaded from repo root (`envDir: "../../"`)
- Path alias: `@` → `packages/client/src/`
- Web workers via comlink plugin

### `assets` — Game Assets (@primodiumxyz/assets, packages/assets/)

- Sprite sheets, VFX, audio; atlas generation via free-tex-packer
- Export commands: `pnpm export:sprites`, `pnpm export:audio`

## Key Patterns

- **MUD Tables as State**: All game state is MUD tables defined in `mud.config.ts`. No separate database. Client syncs state via MUD's indexer/sync stack.
- **System → Library pattern**: Systems validate access and call libraries for logic. Libraries are stateless and read/write MUD tables directly.
- **Prototype configs**: Game entity definitions (buildings, units, resources) are in `config/prototypeConfig.ts`, compiled to Solidity via `ts/prototypes/` codegen.
- **Entity encoding**: `LibEncode.sol` and `src/Keys.sol` define how entities are encoded as `bytes32` keys for MUD tables.
- **Subsystem access control**: Subsystems prefixed with `S_` have `openAccess: false` in MUD config. They're internal-only, called by other systems via the World.

## Environment Setup

Copy `.env.example` to `.env` at repo root. Required vars:

- `PRI_CHAIN_ID` — `"dev"` for local, or a deployed chain ID
- `PRI_DEV` — `"true"` enables dev tools and cheat codes
- `PRI_DEV_PKEY` — Private key for local anvil deployment
- `PRI_WALLETCONNECT_PROJECT_ID` — For WalletConnect integration

Contracts deployer key goes in `packages/contracts/.env`.
